# Handoff: Saya page speed + chat architecture

Context for whoever picks this up next (Codex or otherwise). This covers what shipped, what was actually verified, and — most importantly — what's still unresolved. The lead dev's original complaint ("3 seconds before content starts rendering") is **not yet closed**. What shipped fixed real, verifiable bugs, but nobody has re-run PageSpeed/Lighthouse against production since, so we don't know how much (if any) of the 3s it actually moved.

## What shipped (PR #119, merged to main, live on production as of 2026-07-01)

1. **Removed 3 redundant client-side fetches on Saya pages.** Before this, `pages/menu/[slug].vue`, `components/saya/SayaBlogIndex.vue`, and `pages/blog/[slug].vue` each fired their own separate `useFetch`/`useAsyncData` call on top of the page's shared `useBootstrap()` payload — a second full round trip on every page load for those three routes. Fixed by extending `server/api/public/sites/[siteId]/bootstrap.get.ts` with new `data=blog` / `data=blogPost` dataset types (mirroring the existing `reviews`/`photos`/`qa` pattern) and having those three surfaces read from `useBootstrap()` instead.
2. **Added a KV read-through cache in front of the bootstrap endpoint's D1 batch query** (`server/utils/bootstrap-cache.ts`, `server/plugins/bootstrap-cache-invalidate.ts`). Reuses the existing `SITE_CACHE` KV namespace that already caches full rendered HTML (`server/plugins/edge-cache.ts`) — that HTML cache explicitly excludes `/api/**`, so client-side SPA navigation (which calls bootstrap directly, never re-renders full HTML) had zero caching in front of it before this. Key shape: `bs~<siteId>~<page>~<location>~<experience>~<menu>~<data>~<blogSlug>~<locale>` (percent-encoded fields, `~` delimiter — note this was revised after my first pass used `-`, which collided when field values themselves contained hyphens like slugs; someone else on the team fixed the key encoding in a follow-up commit, `5204697`, along with switching the invalidation purge from fire-and-forget `waitUntil` to an awaited call — check that commit if the cache key format matters to your work).
3. **Invalidation** is wired two ways: an `afterResponse` Nitro hook purges `bs-<siteId>-*` KV keys after any successful non-GET request to `/api/editor/sites/[siteId]/**` (covers all dashboard editor mutations with one hook, no per-route edits), and an explicit call in `server/api/mcp.post.ts` alongside the existing HTML-cache purge (covers MCP tool mutations and ChowBot, which routes through the same MCP tool handlers).
4. **Fixed a regression caught by E2E smoke before merge**: moving blog-detail onto `useBootstrap()` initially dropped the blocking fetch that let the page throw a real HTTP 404 for a missing post (search engines need a real 404, not a 200 "not found" page — there's an E2E test specifically for this, `search-engine-like referrer preserves tenant 404 status and URL`). Fixed in `pages/blog/[slug].vue` by awaiting a dedicated `useAsyncData` call against the same bootstrap key/URL the layout uses (Nuxt's key-based dedup still collapses it to one request), throwing before render if the post is missing.

## What was actually verified (and how)

- Local: typecheck/lint clean on full tree. Manually confirmed cache MISS → HIT (~8s cold → ~20ms) on repeat bootstrap requests via local dev + local KV (`wrangler kv key list --local`). Confirmed a real dashboard PATCH (via dev-login session) purges the bootstrap cache immediately. Confirmed preview-authorized requests and preview/staging hosts skip the cache (never populate or read it).
- Production (read-only, `www.potteryhousekrabi.com` — a real customer): confirmed `/blog/missing-post` returns a real 404. Confirmed the bootstrap cache is genuinely populated with well-formed, fresh-TTL JSON in the production `SITE_CACHE` KV namespace (checked directly via `wrangler kv key get --remote`) for `home`, `experiences`, and `experiences/[slug]` requests.
- **Not verified**: cache invalidation on a real production edit. Attempted this against `site-demo` using the user's session cookie, but hit a dead end — see below.

## The actual unresolved problem: is the site still slow?

Nobody has re-run PageSpeed Insights or a Lighthouse trace against production since this shipped. The original complaint from the lead dev ("crouton") was specifically about **Largest Contentful Paint / time-to-first-content**, not request count in the abstract. Two things are worth knowing before further architectural changes:

1. **Was request-waterfall actually the dominant cost?** An earlier audit this session (before any of the above shipped) found that the "single bootstrap JSON → props" pattern was *already* the dominant architecture across the Saya theme — only these 3 pages violated it. That means the ceiling on how much this PR could improve raw TTFB/FCP was always going to be modest: saving one round trip on 3 out of ~15 page types, not fixing a site-wide waterfall.
2. **Untested alternative hypotheses from this session**, in rough order of suspected impact, none confirmed or ruled out:
   - Cold Worker start / D1 query latency on the *first* (cache-miss) request — this is what the KV cache is supposed to hide, but nobody's measured the actual cache hit rate in production or the cold-path latency with a real trace.
   - Client-side hydration weight — Nuxt/Vue ships a full framework runtime + component code for hydration on every page, unlike e.g. Astro (which the lead dev referenced via EmDash, an Astro-based CMS) which ships near-zero JS by default. This is a *architecture-level* difference, not a caching problem, and nothing in this session touched it.
   - Render-blocking assets (fonts, CSS) delaying first paint even when the HTML itself streams fast.
   - `server/middleware/zz-pageview-tracking.ts` and `server/middleware/tenant-resolution.ts` — confirmed cheap (single non-blocking `waitUntil` insert, single `queryFirst` per request respectively) in earlier analysis, unlikely culprits but not measured under real production load.

**Recommended next step for whoever picks this up**: get one real Lighthouse/PageSpeed trace against `www.potteryhousekrabi.com` (or another live tenant) post-deploy, and look at the actual waterfall/breakdown (TTFB vs. render-blocking resources vs. TBT/hydration cost) before doing more architectural work. Don't guess again — the lead dev's original frustration was partly that guesses ("maybe you did it, maybe you didn't") kept substituting for measurement.

## Small things worth knowing

- Local dev tenant testing on this codebase does NOT work via plain `localhost` — it's hardcoded as a platform host (see `server/utils/tenant-hosts.ts`, deliberate per `CLAUDE.md`'s multi-tenancy rules). Use `curl -H "Host: <subdomain>.localhost:<port>" http://localhost:<port>/...` — `.localhost` resolves to loopback automatically and the tenant-resolution middleware has an explicit `hostname.includes('.localhost')` branch for exactly this.
- `AGENTS.md` has some unexplained drift from `CLAUDE.md` (missing a few recent doc sections) that predates this session — not touched, not investigated, just flagged as existing.
