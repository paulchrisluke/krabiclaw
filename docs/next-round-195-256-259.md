# Delivery plan: issues #195, #256, and #259

This document is the execution contract for the next coordinated round of work targeting `staging`.

Tracked issues:

- #195 — end-customer/guest account model
- #259 — global OG image and social SEO refactor
- #256 — template index and Saya/Blawby detail pages

The objective is to complete all three without creating three competing review streams, without duplicating shared SEO work, and without spending agent time waiting or continuously polling during CodeRabbit's cooldown window.

## Pull-request topology

Create one integration branch and one ready-for-review integration PR:

- `work/next-round-195-256-259` → `staging`

The integration PR is the only PR marked ready for review while implementation is in progress. It tracks all three issues and becomes the single place for final CodeRabbit review, CI, smoke-test evidence, and release readiness.

Create three implementation branches and keep their PRs in draft:

1. `work/259-global-og-seo` → `work/next-round-195-256-259`
2. `work/195-guest-account-model` → `work/next-round-195-256-259`
3. `work/256-template-marketplace` → `work/259-global-og-seo` initially

The #256 PR is intentionally stacked on #259 because #256 must consume the global SEO/OG contract rather than inventing route-specific metadata or image behavior. After #259 merges into the integration branch, retarget #256 to `work/next-round-195-256-259` before merging it.

Draft PRs are implementation workspaces, not independent review queues. Do not request CodeRabbit review on draft PRs. Keep the integration PR current by merging completed draft PRs into the integration branch.

## Ownership boundaries

### #259 — global OG/social SEO foundation

Owns:

- normalized global SEO/social payload contract
- shared metadata composer
- shared OG image URL, cache, rendering, response, and fallback pipeline
- platform, Saya, and Blawby renderer registration
- complete Open Graph and Twitter metadata output
- image validation, metadata, cache-key, renderer, and response tests

May change existing SEO helpers, but platform and tenant helpers must become thin adapters over the same shared contract.

Must not build template marketplace pages from #256.

### #256 — templates index and detail pages

Owns:

- `/templates` index
- `/templates/saya`
- `/templates/blawby`
- canonical public template marketing metadata
- approved Blawby demo URL: `https://ncls.krabiclaw.com`
- template navigation, sitemap, schema, search metadata, and route tests
- broader platform copy audit called for by the issue

Must consume #259's shared metadata and OG image pipeline. It may supply template-specific payload data and renderer selection, but it must not add a parallel SEO helper, fallback chain, or image endpoint.

### #195 — guest account architecture and implementation

Owns:

- an ADR or equivalent documented decision before schema/auth work
- end-customer identity model
- linking authenticated users to tenant-scoped `customers` records
- customer-facing post-login destination and routing
- signup/linking safeguards
- schema, API, auth, and regression tests required by the adopted model

Default architectural direction unless the issue is updated:

- guest users do not receive `organization` or `member` rows
- one Better Auth user may link to many tenant-scoped `customers` rows
- guest/customer UX is separate from `/dashboard/{orgSlug}`
- linking must require verified identity and must not claim unrelated customer records by unverified email alone
- existing tenant-operator signup, invitation, billing, and dashboard behavior must remain unchanged

Avoid unrelated template and OG refactors in this branch.

### Shared account-surface integration contract

#195 owns `server/utils/account-surface.ts` and its authenticated/guest account-resolution API. #259 may consume that API but must not change its authentication, D1 binding, or visibility semantics. Both authenticated tenant operators and authenticated guests must retain their existing account resolution; unauthenticated and unavailable-database results must remain explicit rather than becoming empty successful responses.

After #195 and #259 merge into the integration branch, resolve any edits to this utility in favor of #195's contract, update #259 consumers to that contract, and run an integration test covering both guest resolution and the pre-existing authenticated account behavior. That test is required before the integration PR merges.

## CodeRabbit review cadence

CodeRabbit has a cooldown window. Treat review as a scheduled batch operation, not a polling loop.

For each review cycle:

1. Read the latest integration-PR review, unresolved threads, and CI status once.
2. Classify every item as actionable, already fixed, obsolete, or intentionally declined.
3. Fix all actionable items in one coherent batch.
4. Run targeted tests plus the required broad checks.
5. Push once and post one concise review-resolution summary.
6. Stop checking. Continue useful implementation, tests, documentation, or another independent branch until the next review window or next active session.

Do not:

- wait idly for the cooldown
- continuously poll PR reviews or checks
- repeatedly trigger CodeRabbit during the cooldown
- spend tokens narrating elapsed time
- make one tiny push per review comment when the comments can be resolved together

The implementation agent should checkpoint branch state and remaining work in PR descriptions/comments so a later session can resume without rescanning the repository.

## CI strategy

The integration PR is non-draft and is the canonical preview deployment and smoke-test target.

Draft implementation PRs should remain draft until merged into the integration branch. Their purpose is isolation, diff review, and conflict control. Run local and CI checks appropriate to their scope, but do not promote them into separate full review cycles.

Before pushing a review batch, run at minimum:

- `yarn typecheck`
- `yarn test:unit`
- relevant focused unit tests
- relevant route, image, auth, or E2E tests for the changed issue

For database changes under #195, follow `CLAUDE.md` exactly:

- edit `server/db/schema.ts`
- generate migrations with `yarn db:generate`
- do not edit historical migrations
- run schema and migration checks
- verify current queries and seed behavior

## Required integration smoke matrix

### Global SEO/OG

- platform homepage produces a valid page-specific card
- `/templates`, `/templates/saya`, and `/templates/blawby` use the global metadata contract
- a Saya homepage and content page produce valid `1200×630` images
- the NCLS Blawby homepage, article page, and service page produce valid `1200×630` images
- generated image URLs return actual decodable image bytes, not HTML or serialized data
- explicit image overrides still win where supported
- all covered pages emit complete Open Graph and Twitter metadata

### Template marketplace

- `/templates` is a published-template index
- `/templates/saya` renders the Saya detail experience
- `/templates/blawby` renders the Blawby detail experience
- the Blawby live-demo link points to `https://ncls.krabiclaw.com`
- unknown or unpublished template slugs return 404
- canonical URLs, schema, sitemap entries, navigation, and search metadata are correct

### Guest accounts

- a guest account can link only to customer records it is authorized to claim
- one guest user can relate to customer records from multiple tenant sites without org membership
- guest login lands on the customer-facing surface, not an operator dashboard
- operator signup, invitation, organization membership, billing, and dashboard routing remain unchanged
- guest users cannot access tenant-management APIs or pages

## Final gate before merging to staging

The integration PR may merge only when:

- all three issue scopes are present in the integration branch
- all child draft PRs are merged or closed
- the integration branch is current with `staging`
- no actionable CodeRabbit comments or unresolved review threads remain
- typecheck, unit tests, guardrails, and PR smoke tests are green
- the integration smoke matrix is recorded in the PR
- migration and seed checks are green if #195 changes schema
- no temporary debug code, duplicate SEO pipeline, or temporary showcase URL remains

After merge to `staging`, confirm the staging deployment and staging E2E workflow are green. Fix staging-only failures on the integration/follow-up branch before considering the three issues complete.

## Token and session discipline

- Never wait inside an agent session for an hourly review window.
- Never continuously poll GitHub.
- Use one status read per review cycle.
- Batch review fixes and pushes.
- Switch to independent work when a branch or review is blocked.
- Leave durable checkpoints in GitHub rather than relying on conversational memory.
- Prefer targeted repository reads over repeated broad scans.
